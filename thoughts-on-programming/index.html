Some Random and Abstract Thoughts on Programming

Tweet
  • Programming is a discipline of modeling the knowledge about certain problem in the terms provided by the programming language.
  • Problem has its own domain vocabulary.
  • Programming language has its own vocabulary.
  • Knowing to program in a certain language means knowing to map the vocabulary of the problem to the vocabulary of the language.
  • Knowing some programming language means knowing all the vocabulary it provides and which type of knowledge it is meanth to represent in the problem domain.
  • External knowledge is modeled in programming language in terms of data and procedures that manipulate data.
  • Every programming language is comprised of primitives, means of combinations and means of abstractions.
  • Means of combination are used to build complex expressions out of existing primitives.
  • Means of abstraction are used to define new primitives in the terms of already existing primitives.
  • Every programming language contains already some built in primitive abstractions that are most commonly used.
  • Means of abstraction allow us to develop the vocabulary of the programming language to be closer or to match the vocabulary of the problem that we are solving.
  • Some programming languages (ex. LISP family) contains small number of primitives, but extremely powerful means of abstraction that allow development of more custom programs that match the vocabulary of the domain better.
  • Abstraction principles can be applied both to data and to procedures that manipulate them.
  • Higher-order procedures are a great mean of procedural abstraction.
  • Defining data abstractions in order to separate the way data is used from the way data is defined is a core concept in programming.
  • Algorithms and data structures is a discipline that explores patterns of usage of data and procedures to model standard, often addressed, problems and to give optimal solutions in terms of time and space consumption.
  • Some processes can not be modeled without having to remember their current state in order to make next step.
  • Having a state have a lot of implications.
  • Purely functional programming languages try to find ways to model all the knowledge about processes without using explicit state (ex. Haskell).
  • Avoiding state is in general a good practice.
  • Most often, programmers that come from imperative languages use state extensivelly.
  • There is a lot of places where usage of state can be avoided if process is modeled properly.
